    Security Integration Guide¶

    The Adyen Issuing SDKs handle sensitive data and functionality. To ensure your integration meets security best practices, follow the steps below.

    Security Checklist¶

    • Prevent Screenshots and Screen Recording
    • Prevent Tapjacking and Overlay Attacks
    • Restrict Backup of Sensitive Data
    • Verify Release Builds Are Not Debuggable
    • Protect Exported Components
    • Secure WebViews
    • Handle Sensitive Data in Memory Carefully
    • ProGuard / R8 Configuration
    • Respond to Tamper Signals

    1. Prevent Screenshots and Screen Recording¶

    Apply FLAG_SECURE to any Activity that displays sensitive SDK screens. This also prevents the screen appearing in the recent apps thumbnail.

    override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)
    window.setFlags(
        WindowManager.LayoutParams.FLAG_SECURE,
        WindowManager.LayoutParams.FLAG_SECURE
    )
    setContentView(R.layout.activity_sensitive)
}

    If you are using Jetpack Compose:

    @Composable
fun SensitiveScreen() {
    val activity = LocalContext.current as Activity
    DisposableEffect(Unit) {
        activity.window.setFlags(
            WindowManager.LayoutParams.FLAG_SECURE,
            WindowManager.LayoutParams.FLAG_SECURE
        )
        onDispose {
            activity.window.clearFlags(WindowManager.LayoutParams.FLAG_SECURE)
        }
    }
    // your content
}

    Note: FLAG_SECURE does not prevent all forms of screen capture on rooted devices. For more information, see the Android documentation on window flags.

    2. Prevent Tapjacking and Overlay Attacks¶

    Tapjacking tricks users into tapping UI elements hidden beneath a malicious overlay. Filter out touch events that have been obscured by another window. For more information see the Android documentation on Tapjacking.

    Apply this to any View handling sensitive interactions (confirm buttons, PIN entry, etc.):

    sensitiveButton.setOnTouchListener { view, event ->
    if (event.flags and MotionEvent.FLAG_WINDOW_IS_OBSCURED != 0) {
        // Touch event obscured by an overlay — reject it
        return@setOnTouchListener true
    }
    view.onTouchEvent(event)
}

    Or apply it globally across your Activity by overriding dispatchTouchEvent:

    override fun dispatchTouchEvent(event: MotionEvent): Boolean {
    if (event.flags and MotionEvent.FLAG_WINDOW_IS_OBSCURED != 0) {
        return false
    }
    return super.dispatchTouchEvent(event)
}

    You can also declare this in XML on individual Views:

    <Button
    android:filterTouchesWhenObscured="true"
    ... />

    3. Restrict Backup of Sensitive Data¶

    Android's auto-backup can inadvertently include sensitive SDK data. Disable backup entirely or use backup rules to exclude sensitive files. See the Android documentation on Backing up app data for details.

    In your AndroidManifest.xml:

    <application
    android:allowBackup="false"
    ... >

    If you need to allow backup for other app data, use backup rules to exclude SDK data (Android 12+):

    <!-- res/xml/backup_rules.xml -->
<?xml version="1.0" encoding="utf-8"?>
<full-backup-content>
    <exclude domain="sharedpref" path="com.yoursdk.prefs" />
    <exclude domain="database" path="yoursdk.db" />
    <exclude domain="file" path="yoursdk/" />
</full-backup-content>

    <!-- AndroidManifest.xml -->
<application
    android:fullBackupContent="@xml/backup_rules"
    android:dataExtractionRules="@xml/backup_rules"
    ... >

    4. Verify Release Builds Are Not Debuggable¶

    Ensure your release build is not debuggable. A debuggable app allows attackers to attach a debugger, inspect memory, and bypass security checks. The Android Gradle Plugin sets isDebuggable = false by default for the release build type, but you should verify it is not accidentally overridden or inherited in custom build types.

    5. Protect Exported Components¶

    Ensure Activities, Services, and Receivers that are not intended for external use are not exported. Any exported component is a potential entry point for attack.

    <!-- Unexported — internal use only -->
<activity
    android:name=".SensitiveActivity"
    android:exported="false" />

<!-- If a component must be exported, restrict with a signature-level permission -->
<activity
    android:name=".DeeplinkActivity"
    android:exported="true"
    android:permission="com.yourapp.permission.DEEPLINK" />

<permission
    android:name="com.yourapp.permission.DEEPLINK"
    android:protectionLevel="signature" />

    6. Secure WebViews¶

    If your integration uses WebViews, apply the following hardening. See the Android Web App Security documentation for a complete overview.

    webView.settings.apply {
    javaScriptEnabled = true // only if strictly required
    allowFileAccess = false
    allowContentAccess = false
    allowFileAccessFromFileURLs = false
    allowUniversalAccessFromFileURLs = false
    setSupportMultipleWindows(true)
    javaScriptCanOpenWindowsAutomatically = false
}

// Remove interfaces not in use
webView.removeJavascriptInterface("searchBoxJavaBridge_")
webView.removeJavascriptInterface("accessibility")
webView.removeJavascriptInterface("accessibilityTraversal")

// Restrict navigation to known hosts
webView.webViewClient = object : WebViewClient() {
    override fun shouldOverrideUrlLoading(view: WebView, request: WebResourceRequest): Boolean {
        val host = request.url.host ?: return true
        return if (host.endsWith("your-trusted-domain.com")) {
            false // allow
        } else {
            true // block
        }
    }
}

    7. Handle Sensitive Data in Memory Carefully¶

    Avoid passing sensitive values (tokens, keys, PINs) as String objects where possible, as strings are immutable and cannot be zeroed out after use. Prefer CharArray or ByteArray so you can explicitly clear them when done.

    fun handlePin(pin: CharArray) {
    try {
        // use pin
    } finally {
        pin.fill('\u0000') // zero out immediately after use
    }
}

    Never log sensitive values, even in debug builds:

    // ❌ Never do this
Log.d("SDK", "User token: $token")

// ✅ Log safe identifiers only
Log.d("SDK", "Token received, length=${token.size}")

    8. ProGuard / R8 Configuration¶

    Ensure minification is enabled in your release builds. The SDK ships with its own consumer ProGuard rules, but verify your setup keeps R8 active:

    buildTypes {
    release {
        isMinifyEnabled = true
        isShrinkResources = true
    }
}

    If you are using any SDK models in your own ProGuard rules, only keep rules that are strictly necessary. Overly broad -keep rules will undo obfuscation and make reverse engineering easier. For detailed documentation on setting up code shrinking and obfuscation, see the Android Shrink, obfuscate, and optimize your app documentation.

    # ❌ Too broad — avoid
-keep class com.example.** { *; }

# ✅ Keep only what your code references directly
-keep class com.example.sdk.MySdk { public *; }

    For advanced security and obfuscation beyond what R8 provides, you may want to consider using a commercial application protection product such as DexGuard, Appdome, or Promon. These tools offer features like string encryption, class encryption, and runtime application self-protection (RASP).

    9. Respond to Tamper Signals¶

    The Adyen Issuing SDKs perform Runtime Application Self-Protection (RASP) checks at sensitive call sites (see the Environment Constraints section of the main README). When a threat is detected (root, hooking, memory scanning, emulator, virtual environment, compromised hardware, or a debugger attached to a release build), the SDK stops performing sensitive work and returns a TamperError result to your code.

    The client-side SDK intentionally does not wipe keys or cached state on tamper detection:

    • Session private keys are hardware-backed (StrongBox / TEE) and cannot be exfiltrated even on a compromised device — the SDK simply refuses to use them.
    • Persisted state is wrapped with those TEE-bound keys and is useless off-device.
    • An attacker capable of bypassing RASP could equally hook out any client-side wipe logic, so "refuse to operate" is a stronger invariant than "try to delete".

    Because of this, server-side session invalidation is your responsibility. When an SDK call returns a TamperError variant, treat it as a cue to terminate the user's session on your backend (logout, session revoke, risk flag) so the session token is refused regardless of what the compromised client does next. Combine this with short-lived session tokens and server-side rate-limiting for defence-in-depth.

    Example for the Session-based Provisioning SDK:

    when (val result = provisioningSession.canProvisionCard(paymentInstrumentId)) {
    is CanProvisionResult.CannotBeProvisioned.TamperError -> {
        // Invalidate the user's session on your backend, then surface a
        // neutral "unable to continue" UI. Do not hint at the reason.
        backend.invalidateSession()
        showGenericErrorAndLogOut()
    }
    // … handle other result variants
    else -> { /* … */ }
}

    The same pattern applies to ProvisionResult.Failure.TamperError returned from provisionCard, and to any TamperError surfaced by the core CardDataSource if you integrate directly against it.

    Do not surface the specific tamper reason to the user — a neutral error message avoids giving attackers feedback on which check triggered.
