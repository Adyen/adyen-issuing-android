    Example App¶

    This example application demonstrates how to integrate both Adyen Issuing Network Token Provisioning SDKs to add cards to Google Wallet from your Android app:

    • The Session-based Provisioning SDK — session-scoped variant where the SDK manages the back-and-forth with Adyen behind a session token.
    • The Offline Provisioning SDK — direct-integration variant where your backend exchanges sdkInput / sdkOutput opaque blobs with Adyen.

    A single Compose screen exercises whichever SDK is selected at runtime via a segmented-button toggle at the bottom of the main screen.

    ⚠️ Reference only — not functional end-to-end. The app runs against in-memory fakes and a placeholder application certificate, so the wallet-add step always fails. See Important notes for what must be replaced for a live integration.

    Session-based Provisioning SDK demo¶

    The integration is implemented in the provisioningsessions/ package and exposed through ProvisioningSessionsRepository:

    1. Embed an application certificate. MainActivity loads it from res/raw/fake_app_cert and passes the input stream into the repository.
    2. Create a session. ProvisioningSession.create(...) is invoked with the certificate, the set of payment-instrument IDs, an activity provider, and a sessionTokenProvider lambda that calls SessionsBackend.fetchToken(paymentInstrumentIds) (your backend would obtain a session token via Adyen's session API).
    3. Check eligibility. session.canProvisionCard(paymentInstrumentId) returns the same CanBeProvisioned / CannotBeProvisioned.* shape as the direct SDK and is mapped onto the shared CardState.
    4. Provision the card. session.provisionCard(paymentInstrumentId, cardDisplayName, cardAddress) runs the wallet flow without further per-step backend involvement.

    FakeSessionsBackend returns a static base64 session token, so the wallet-add step also fails by design with mock data.

    Offline Provisioning SDK demo¶

    The integration is implemented in the provisioningsdk/ package and exposed through ProvisioningSdkRepository:

    1. Fetch activation data. SdkBackend.requestCardActivation(paymentInstrumentId) returns the opaque sdkInput blob your backend would receive from Adyen (GET .../paymentInstruments/{id}/networkTokenActivationData).
    2. Initialise the SDK. CardProvisioning.create(sdkInput, activityProvider) builds a CardProvisioning instance.
    3. Check provisioning eligibility. canProvision() resolves to CanBeProvisioned, AlreadyExistsInWallet, an ApiError / Error / UnknownFailure, and is mapped onto the shared CardState.
    4. Provision the card. provision(...) is invoked with an opcProvider lambda that re-enters the backend (requestOpaquePaymentCardData(paymentInstrumentId, sdkOutput)) using the sdkOutput to fetch the matching Opaque Payment Card data; the lambda returns it to the SDK, which completes the wallet add.

    FakeSdkBackend returns a static base64 sdkInput and a placeholder OPC string, so the wallet-add step intentionally fails — the demo wires the call sequence and result mapping correctly, but the data is not valid for live provisioning.

    Key components¶

    File Role
    MainActivity.kt Sets up the Compose UI; loads R.raw.fake_app_cert; provides the Activity reference to both SDKs.
    viewmodel/MainViewModel.kt Holds both repositories; dispatches getCardState(...) and provisionCard(...) to whichever SDK is selected.
    viewmodel/MainViewModelFactory.kt Wires the fakes (FakeSdkBackend, FakeSessionsBackend) into the repositories.
    data/ProvisioningMethod.kt Enum with SDK and SESSIONS entries; controls runtime dispatch.
    data/CardState.kt UI-facing state union (Loading, NotAddedToWallet, Provisioning, AddedToWallet, NotSupported, Disabled, Error) shared across both demos.
    repository/ProvisioningRepository.kt Common interface implemented by both per-SDK repositories.
    provisioningsdk/repository/ProvisioningSdkRepository.kt Offline Provisioning SDK integration.
    provisioningsessions/repository/ProvisioningSessionsRepository.kt Session-based Provisioning SDK integration.
    provisioningsdk/backend/SdkBackend.kt + provisioningsdk/fake/FakeSdkBackend.kt Backend interface + in-memory fake for the direct SDK flow.
    provisioningsessions/backend/SessionsBackend.kt + provisioningsessions/fake/FakeSessionsBackend.kt Backend interface + in-memory fake for the Sessions flow.
    ui/MainScreen.kt Compose UI; the segmented-button row at the bottom toggles between SDK and SESSIONS.

    Getting started¶

    1. Get the Google Push Provisioning API artifact. Both SDKs depend on com.google.android.gms:play-services-tapandpay, which is not on any public Maven repository. Follow the family-level Google Push Provisioning API Access guide.
    2. Clone the repository.
    3. Open the project in Android Studio.
    4. Build and run the example module on an Android device or emulator with Google Play Services. Use the segmented button at the bottom of the main screen to switch between the SDK and Sessions demos.

    Important notes¶

    • Mock data. Tapping "Add to Google Wallet" produces an error in either mode — the mocked sdkInput / OPC / session token are syntactically valid placeholders but are not accepted by Adyen's or Google's live infrastructure.
    • Backend integration. FakeSdkBackend and FakeSessionsBackend are demo-only. In production you must implement real backend services that proxy to Adyen's Network Token APIs (for the SDK flow) or obtain session tokens via Adyen's session API (for the Sessions flow).
    • Application certificate. The bundled res/raw/fake_app_cert is a placeholder. The Sessions SDK requires a real application certificate provisioned by Adyen for production use.
    • Security. Never hardcode sensitive card data, OPC payloads, or session tokens in your application; always fetch them from your backend over an authenticated channel.
    • App architecture. The architecture, project structure, coding conventions, and UI design here are illustrative only — they don't represent required or preferred patterns from Adyen.
