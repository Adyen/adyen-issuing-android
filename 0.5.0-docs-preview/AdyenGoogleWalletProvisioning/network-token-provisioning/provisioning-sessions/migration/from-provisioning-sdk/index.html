    Migrating from the Offline Provisioning SDK to the Session-based Provisioning SDK¶

    Guidance for moving an existing integration from the Offline Provisioning SDK to the Session-based Provisioning SDK.

    The example app contains both integrations side-by-side under provisioningsdk/ and provisioningsessions/, sharing a single ProvisioningRepository interface — a useful concrete reference for the diff described below.

    Coming from a < 0.4.0 Offline Provisioning SDK?

    You don't need to do the pre-0.4 migration first. The "Before" snippets on this page show the post-0.4.0 Offline Provisioning SDK API so the diff to the Session-based Provisioning SDK is small, but if your existing integration is on < 0.4.0 you can replace it with the Session-based flow directly — none of the pre-0.4.0 createSdkOutput() / manual OPC orchestration carries over either way. Use the Session-based Provisioning SDK — Getting Started as your primary reference and treat the "Before" snippets here as illustrative of what the Offline Provisioning SDK call site looks like, not as a prerequisite your code must already match.

    Why migrate¶

    Session-based provisioning consolidates several integration concerns that the Offline Provisioning SDK delegates to the backend:

    • Smaller backend surface area. A single session-token endpoint replaces the two-leg networkTokenActivationData exchange (GET for sdkInput, POST with sdkOutput for OPC). Once the session token is issued, the SDK handles the remainder of the provisioning protocol with Adyen internally.
    • Multi-card sessions. A single ProvisioningSession covers a Set<String> of payment-instrument IDs, so flows that present several cards at once (e.g. a wallet management screen) incur the session-creation cost only once.
    • Tamper-detection signals. The session pipeline surfaces device-tamper outcomes alongside the standard provisioning results, providing an integrated signal in client error handling rather than a separate workflow.
    • Simpler client lifecycle. No OpcProvider callback to integrate with the backend; no per-step opaque-payload handling in application code.

    The trade-off is reduced step-by-step control. If there are integration-specific reasons to stage sdkInput / sdkOutput through application logic between canProvision and provision, the Offline Provisioning SDK remains the appropriate choice.

    APP->>SDK: provisionCard(paymentInstrumentId, ...)
    activate SDK
    SDK-->>APP: ProvisionResult.Success
    deactivate SDK

    Maven coordinate com.adyen.issuing.provisioning:provisioning-sdk com.adyen.issuing.provisioning:provisioning-sessions
    Primary type CardProvisioning (one instance per card) ProvisioningSession (one instance per card set)
    Factory CardProvisioning.create(sdkInput, activityProvider) ProvisioningSession.create(context, appCertInputStream, paymentInstrumentIds, activityProvider, sessionTokenProvider, environment = LIVE)
    Eligibility check canProvision(): CanProvisionResult canProvisionCard(paymentInstrumentId): CanProvisionResult
    Provision provision(cardDisplayName, cardAddress, opcProvider): ProvisionResult provisionCard(paymentInstrumentId, cardDisplayName, cardAddress): ProvisionResult

    Environment selector Not exposed ProvisioningEnvironment (LIVE, BETA, TEST)
    Result types package com.adyen.issuing.mobile.provisioning.client.results.* com.adyen.issuing.mobile.provisioning.sessions.results.*

    Activity provider () -> Activity Same signature
    Experimental opt-in @AdyenExperimentalApi Same annotation

    The two SDKs use distinct results packages with different concrete variants, so any when expression that is exhaustive over the SDK's CanProvisionResult / ProvisionResult / CardProvisioningCreateResult branches will need to be updated to cover the Sessions counterparts. See Result-type expansion below.

    Backend changes¶

    What goes away¶

    The two networkTokenActivationData calls used by the direct SDK are no longer invoked from the client:

    • GET .../paymentInstruments/{id}/networkTokenActivationData — previously returned sdkInput.
    • POST .../paymentInstruments/{id}/networkTokenActivationData — previously returned the OPC payload in exchange for sdkOutput.

    Once the migration is complete and the SDK consumers are removed, the corresponding backend endpoints can be decommissioned (or repurposed for any non-SDK use cases).

    What replaces them¶

    A single endpoint on your backend that creates an Adyen session token for the set of payment-instrument IDs the user is about to operate on. Its response — a base64-encoded session token issued by Adyen's session API — is what your sessionTokenProvider lambda returns to the SDK.

    The lambda signature is:

    sessionTokenProvider: suspend (Set<String>) -> String

    It receives the same paymentInstrumentIds set you passed to ProvisioningSession.create(...) and is expected to return the session token string.

    Your backend obtains the token by calling Adyen's session API:

    • Live — POST https://authe-live.adyen.com/authe/api/v1/sessions
    • Test — POST https://authe-test.adyen.com/authe/api/v1/sessions

    Client code changes¶

    The example app demonstrates both call sequences side-by-side; the snippets below summarise the differences.

    Before — Offline Provisioning SDK¶

    // Backend call 1: GET networkTokenActivationData -> sdkInput
val sdkInput: String = sdkBackend.requestCardActivation(paymentInstrumentId).sdkInput

val createResult = CardProvisioning.create(
    sdkInput = sdkInput,
    activityProvider = { hostingActivity },
)
val cardProvisioning = (createResult as CardProvisioningCreateResult.Success).cardProvisioning

when (cardProvisioning.canProvision()) {
    is CanProvisionResult.CanBeProvisioned -> { /* show Add to Wallet */ }
    is CanProvisionResult.CannotBeProvisioned.AlreadyExistsInWallet -> { /* hide button */ }
    is CanProvisionResult.CannotBeProvisioned.ApiError,
    is CanProvisionResult.CannotBeProvisioned.Error,
    is CanProvisionResult.CannotBeProvisioned.UnknownFailure -> { /* show error */ }
}

val provisionResult = cardProvisioning.provision(
    cardDisplayName = "John Doe",
    cardAddress = CardAddress(),
    opcProvider = { paymentInstrumentId, sdkOutput ->
        // Backend call 2: POST networkTokenActivationData with sdkOutput -> OPC
        sdkBackend.requestOpaquePaymentCardData(paymentInstrumentId, sdkOutput).sdkInput
    },
)

    After — Session-based Provisioning SDK¶

    val createResult = ProvisioningSession.create(
    context = applicationContext,
    appCertInputStream = resources.openRawResource(R.raw.app_cert),
    paymentInstrumentIds = setOf(paymentInstrumentId),
    activityProvider = { hostingActivity },
    sessionTokenProvider = { paymentInstrumentIds ->
        // Single backend call: the endpoint requests a token from Adyen's session API
        sessionsBackend.fetchToken(paymentInstrumentIds)
    },
)
val session = (createResult as ProvisioningSessionCreateResult.Success).session

when (session.canProvisionCard(paymentInstrumentId)) {
    is CanProvisionResult.CanBeProvisioned -> { /* show Add to Wallet */ }
    is CanProvisionResult.CannotBeProvisioned.AlreadyExistsInWallet -> { /* hide button */ }
    // ... see Result-type expansion below for the additional variants to handle
}

val provisionResult = session.provisionCard(
    paymentInstrumentId = paymentInstrumentId,
    cardDisplayName = "John Doe",
    cardAddress = CardAddress(),
)

    The CardAddress import does not change — it is defined in the Offline Provisioning SDK's client.data package and re-used unchanged by the Session-based Provisioning SDK.

    Result-type expansion¶

    The Session-based result types intentionally expose a wider failure surface than the Offline Provisioning SDK because operational concerns the Offline Provisioning SDK delegates to your backend (token-fetch errors, certificate validation, tamper detection, transport / serialization failures) become observable inside the SDK once it manages the session.

    Concrete additions to cover when updating exhaustive when expressions:

    ProvisioningSessionCreateResult.Failure — replaces CardProvisioningCreateResult.Failed:

    • ApplicationCertificateError — the bytes you supplied could not be parsed as an X.509 certificate.
    • Error(message: String?) — generic creation failure with optional reason.

    GooglePayNotSupported and InvalidSdkInput from the SDK no longer occur at session creation: GooglePayNotSupported moves to CanProvisionResult.CannotBeProvisioned, and InvalidSdkInput no longer exists because there is no opaque input payload.

    Sessions CanProvisionResult.CannotBeProvisioned — supersedes the SDK's variants:

    • All four SDK variants (AlreadyExistsInWallet, ApiError, Error, UnknownFailure) plus a GooglePayNotSupported branch that previously surfaced at create-time.
    • New: DecryptError, EncryptError, HttpError, SdkCertificateError, InvalidApplicationCertificate, PaymentInstrumentNotAllowed, SerializationError, TokenProviderError, TamperError.

    Sessions ProvisionResult.Failure — supersedes the SDK's variants:

    • New: AlreadyProvisioned, Canceled, DecryptError, EncryptError, Error, HttpError, SdkCertificateError, InvalidApplicationCertificate, PaymentInstrumentNotAllowed, ProvisioningInProgress, SerializationError, TokenProviderError, ProvisioningSdkFailure, TapAndPayApiFailure, TapAndPayNotAvailable, TamperError.

    Most of these can be mapped to a small set of UI states for typical applications (retryable transport error, non-retryable validation error, terminal "card is already in wallet"), but the Kotlin compiler will require each branch to be addressed explicitly.

    What remains unchanged¶

    • Google Push Provisioning API allowlisting. Both SDKs depend transitively on com.google.android.gms:play-services-tapandpay and the same allowlisting steps apply. See Google Push Provisioning API Access.
    • CardAddress. Same fields, same package, no migration required.
    • activityProvider: () -> Activity. The lambda is invoked in the same way; pass the host Activity as before.
    • @AdyenExperimentalApi. Both SDKs require the same opt-in annotation at call sites.
    • Minimum Android API and compileSdk requirements. Unchanged at API 26 / compileSdk 35.

    Step-by-step migration¶

    1. Add the new dependency alongside the existing one so the migration can be shipped incrementally:

      dependencies {
    implementation("com.adyen.issuing.provisioning:provisioning-sdk:0.5.0-docs-preview") // existing
    implementation("com.adyen.issuing.provisioning:provisioning-sessions:0.5.0-docs-preview") // new
}

    2. Embed the application certificate. Place the certificate provided by Adyen in res/raw/ (or another readable location) so it can be passed to ProvisioningSession.create(...) as an InputStream.

    3. Implement the session-token backend endpoint that proxies to Adyen's session API and returns the issued token. Integrate it into the client behind a sessionTokenProvider lambda.

    4. Introduce a new repository or use-case that encapsulates the Session-based flow without disturbing the existing Offline Provisioning SDK code path. The example app's ProvisioningSessionsRepository (alongside the existing ProvisioningSdkRepository implementing the same ProvisioningRepository interface) is the reference layout.

    5. Switch call sites incrementally. A feature flag or runtime toggle (the example app uses a segmented button bound to a ProvisioningMethod enum) allows routing a percentage of traffic through the Session-based Provisioning SDK while leaving fallback to the Offline Provisioning SDK in place.

    6. Update the error mapping to cover the expanded CanProvisionResult and ProvisionResult branches. Treat the new *Error variants conservatively — retry where transport-related; otherwise surface the failure and abort.

    7. Decommission the old API surface once the Session-based Provisioning SDK is rolled out: remove the Offline Provisioning SDK dependency, delete the OpcProvider integration code, and retire the two networkTokenActivationData backend endpoints if no other consumers remain.
