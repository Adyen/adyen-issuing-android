    Getting Started¶

    Integrate the Adyen Session-based Provisioning SDK into your Android application.

    Prerequisites¶

    Before you begin:

    • Your application has a minSdk of 26 (Android 8.0) or higher.
    • Your application has a compileSdk of 35 or higher.
    • An API credential with the Network Token: Create role.
    • Your app ID is allowlisted by Google for the Push Provisioning API — see Google Push Provisioning API Access (shared across both SDKs in this family).

    In addition, the Session-based SDK requires an application certificate provided by Adyen — see below.

    Adyen Session-based Provisioning SDK application certificate¶

    The application certificate is specific to your application and lets the SDK verify the host application at runtime. You can request it from your Adyen Project Operations Manager or Implementation Manager. You will need to provide your Android application's applicationId (look for the applicationId property in your main app module's build.gradle.kts).

    If your applicationId ever changes, request a new certificate for the new id — the existing one will not match.

    The certificate is provided as a file with a .cer extension. You will embed it in your app in step 1.

    1. Embed the Certificate¶

    Add the .cer file Adyen provided to your app module's res/raw resource directory (create it if you do not already have one). At runtime, load it into an InputStream:

    // Load the certificate data from a raw resource named `app_cert`.
val appCertInputStream = context.resources.openRawResource(R.raw.app_cert)

    You will pass this InputStream into ProvisioningSession.create() in step 4.

    2. Fetch a Session Token¶

    Each provisioning attempt is scoped to a session token your backend obtains from the Adyen API. From your backend, perform:

    URL — https://test.adyen.com/authe/api/v1/sessions (use the live host for production).

    Headers

    • Content-Type: application/json
    • X-API-Key: YOUR_API_KEY

    Body

    {
    "allowOrigin": "https://www.adyen.com",
    "product": "platform",
    "policy": {
        "resources": [
            { "type": "paymentInstrument", "paymentInstrumentId": "PI..." }
        ],
        "roles": ["Network Token: Create"]
    }
}

    Have your backend return the resulting session token to your Android app on demand. The SDK will request a token via the sessionTokenProvider callback you provide in step 4.

    3. Add the Session-based Provisioning SDK Dependency¶

    The SDK is published to Maven Central. Add the dependency to your app module's build.gradle.kts:

    dependencies {
    implementation("com.adyen.issuing.provisioning:provisioning-sessions:0.5.0-docs-preview")
}

    Or the equivalent version catalog entry.

    Tip

    Always check Maven Central for the latest version before integrating.

    4. Initialize the Provisioning Session¶

    Create a ProvisioningSession via the static factory:

    public fun create(
    context: Context,
    appCertInputStream: InputStream,
    paymentInstrumentIds: Set<String>,
    activityProvider: () -> Activity,
    sessionTokenProvider: suspend (Set<String>) -> String,
    environment: ProvisioningEnvironment = ProvisioningEnvironment.LIVE,
): ProvisioningSessionCreateResult

    Parameter rundown:

    • context — application or activity context.
    • appCertInputStream — the application-certificate InputStream from step 1.
    • paymentInstrumentIds — the payment instruments that may be provisioned during this session.
    • activityProvider — function returning the host Activity (typically the Activity hosting your card-provisioning screen, or your single-Activity app's main Activity).
    • sessionTokenProvider — suspend function the SDK invokes when it needs a session token (see step 2).
    • environment — defaults to ProvisioningEnvironment.LIVE. Switch to ProvisioningEnvironment.TEST for the Adyen test environment.

    Note

    There is also an overload that accepts a java.security.cert.X509Certificate directly if you would prefer to manage the certificate as an in-memory object rather than a stream.

    A typical wiring up looks like:

    suspend fun fetchSessionTokenFromBackend(paymentInstrumentIds: Set<String>): String {
    val body = buildSessionTokenRequestBody(paymentInstrumentIds) // see step 2
    val response = httpClient.post("yourBackend/sessionToken", body)
    if (!response.isSuccessful()) {
        // Throw — the SDK will surface the failure as a Failure result to your caller.
        throw IllegalStateException("Could not fetch session token: ${response.code}")
    }
    return response.bodyText()
}

val result = ProvisioningSession.create(
    context = applicationContext,
    appCertInputStream = appCertInputStream,
    paymentInstrumentIds = setOf("PI...01", "PI...02"),
    activityProvider = { provisioningActivity },
    sessionTokenProvider = { paymentInstrumentIds ->
        fetchSessionTokenFromBackend(paymentInstrumentIds)
    },
)

val provisioningSession = when (result) {
    is ProvisioningSessionCreateResult.Success ->
        result.session
    is ProvisioningSessionCreateResult.Failure.ApplicationCertificateError ->
        throw IllegalStateException("Invalid application certificate")
    is ProvisioningSessionCreateResult.Failure.Error ->
        throw IllegalStateException("Unexpected initialization error")
}

    The Failure variants distinguish between a malformed / mis-issued application certificate and any other unexpected initialization error.

    5. Check if a Card Can Be Provisioned¶

    Before showing an "Add to Google Wallet" affordance, ask the session whether each card can be added to the user's wallet:

    when (val result = provisioningSession.canProvisionCard(paymentInstrumentId)) {
    // Card is ready to be added to the wallet.
    is CanProvisionResult.CanBeProvisioned ->
        showAddToWalletButton()
    // Card has already been added to this device's wallet.
    is CanProvisionResult.CannotBeProvisioned.AlreadyExistsInWallet ->
        hideAddToWalletButton()
    // Card cannot be added for some other reason.
    else ->
        showCannotBeProvisionedReason(result)
}

    canProvisionCard() is a suspend function — call it from a coroutine. Treat the result as advisory and re-check if the user revisits the screen later.

    6. Provision the Card¶

    When the user taps "Add to Google Wallet", drive Google's native provisioning flow:

    val provisionResult = provisioningSession.provisionCard(
    paymentInstrumentId = paymentInstrumentId,
    cardDisplayName = "John Doe",
    cardAddress = CardAddress(),
)

    provisionCard() is a suspend function and presents Google's native "Add to Wallet" UI as part of its execution. Call it from the Activity lifecycle (e.g. inside lifecycleScope.launch { ... } triggered by the user's tap).

    Note

    Passing an empty CardAddress() defers the address-entry step to Google Wallet's native flow — the user will be prompted to enter their address during provisioning. If you already have a verified address for the cardholder, populate the CardAddress fields to skip that step.

    7. Handle the Final State¶

    provisionCard() returns a ProvisionResult once the Google Wallet flow completes (successfully or otherwise). Branch on it to update your UI:

    when (provisionResult) {
    is ProvisionResult.Success ->
        handleProvisioningSuccess(provisionResult.tokenId)
    is ProvisionResult.Failure ->
        handleProvisioningFailure(provisionResult)
}

    The Failure hierarchy distinguishes error categories such as user cancellation, network problems, and the tamper signal covered next.

    8. Respond to TamperError¶

    Both canProvisionCard() and provisionCard() may return a TamperError variant when the SDK's Runtime Application Self-Protection (RASP) checks detect that the device has been compromised — root, hooking, memory scanning, emulator, virtual environment, compromised hardware, or a debugger attached to a release build.

    When you receive a TamperError, treat it as a signal to invalidate the user's session on your backend. The SDK halts sensitive work on the client side, but it does not terminate the server-side session — that responsibility is delegated to you so the response composes with your wider fraud controls.

    when (val result = provisioningSession.canProvisionCard(paymentInstrumentId)) {
    is CanProvisionResult.CannotBeProvisioned.TamperError -> {
        // Revoke the session on your backend so the session token is refused
        // regardless of what the compromised client does next, then surface a
        // neutral error to the user.
        backend.invalidateSession()
        showGenericErrorAndLogOut()
    }
    // … handle other result variants
    else -> { /* … */ }
}

when (val result = provisioningSession.provisionCard(paymentInstrumentId, cardDisplayName, cardAddress)) {
    is ProvisionResult.Failure.TamperError -> {
        backend.invalidateSession()
        showGenericErrorAndLogOut()
    }
    // … handle other result variants
    else -> { /* … */ }
}

    Do not surface the specific tamper reason to the user — a neutral error message avoids giving attackers feedback on which check triggered. The tamperCode on the result is intended for reporting to Adyen support, not for end-user display.

    See Respond to Tamper Signals in the Security Guide for the full rationale, including why the client-side SDK does not self-wipe keys or cached state on detection.
