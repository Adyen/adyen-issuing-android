    Migrating from SDK versions prior to 0.4.0¶

    Guidance for upgrading existing integrations from provisioning-sdk 0.3.x and earlier to 0.4.0+.

    Migrating straight to the Session-based Provisioning SDK?

    If you are on < 0.4.0 and your end goal is the Session-based Provisioning SDK, you can skip this migration entirely. The Session-based Provisioning SDK uses provisioning-sdk 0.4.0+ internally and exposes a different client API — the createSdkOutput() / OpcProvider / direct-SDK distinctions never surface in your code. Follow the Session-based Provisioning SDK — Getting Started instead. This page is for integrations that intend to stay on the Offline Provisioning SDK.

    Why upgrade¶

    The primary reason to move to 0.4.0+ is forward compatibility with Google's Unified Push Provisioning (UPP).

    provisioning-sdk — including the current 0.4.0+ line — is still implemented on top of Google's legacy Tap and Pay push-provisioning API. Google has marked that legacy API as deprecated and is replacing it with Unified Push Provisioning, which supports saving cards to a user's Google account rather than only to the specific device that ran the provisioning flow. Adyen will swap the SDK's internals over to UPP in a future release; that release will be designed as a drop-in version bump for clients already on the 0.4.0+ client API. Clients still on < 0.4.0 at that point will face an additional breaking-change migration on top of the SDK upgrade, because the pre-0.4.0 createSdkOutput() + provision(sdkInput, …) API cannot be preserved on top of UPP. Migrating now decouples your code from that future transition.

    The 0.4.0+ API is built around a single OpcProvider callback that the SDK invokes when it requires the Opaque Payment Card (OPC) data, rather than requiring the application to orchestrate that exchange. Pre-0.4.0 integrations call createSdkOutput(), send the resulting sdkOutput to the backend to retrieve the OPC value from Adyen, and only then call provision(sdkInput, cardDisplayName, cardAddress). From 0.4.0 you pass an OpcProvider to a single provision(cardDisplayName, cardAddress, opcProvider) call, and the SDK performs the round-trip internally.

    Secondary benefits picked up along the way:

    • Fewer steps at the call site. One provision(cardDisplayName, cardAddress, opcProvider) call replaces the two-stage createSdkOutput → backend call → provision(sdkInput, …) sequence.
    • Backend integration is unchanged. The two networkTokenActivationData endpoints (GET for sdkInput, POST for OPC) still carry the same payloads — only the timing of their invocation from the client changes.
    • Unified error handling. Failures inside the OPC retrieval surface through the single ProvisionResult returned from the new overload, rather than being split across CreateSdkOutputResult and ProvisionResult.

    Breaking changes¶

    The two pre-0.4.0 entry points are now deprecated. Both still compile but emit warnings, and both will be removed in a future release.

    API Status from 0.4.0 Replacement
    CardProvisioning.createSdkOutput(): CreateSdkOutputResult @Deprecated Not needed — the new provision(...) overload performs the equivalent step internally.
    CardProvisioning.provision(sdkInput, cardDisplayName, cardAddress): ProvisionResult @Deprecated (ReplaceWith("provision(cardDisplayName, cardAddress, opcProvider)")) provision(cardDisplayName, cardAddress, opcProvider): ProvisionResult
    CreateSdkOutputResult (sealed interface and all Success / Failure.* variants) @Deprecated Failure cases that previously surfaced here now flow through ProvisionResult.Failure.

    The IDE quick-fix on the deprecated provision(...) overload offers the structural rewrite via the ReplaceWith hint, but you must additionally supply an OpcProvider implementation that performs the existing OPC backend call.

    OpcProvider interface¶

    OpcProvider is a fun interface (single abstract method, so it accepts a SAM lambda):

    public fun interface OpcProvider {
    public suspend fun fetchOpc(
        paymentInstrumentId: String,
        sdkOutput: String,
    ): String
}

    The SDK invokes fetchOpc once per provision(...) call, passing the paymentInstrumentId of the card being provisioned and a newly issued sdkOutput. The implementation must return the base64-encoded OPC string from the backend. Exceptions thrown from fetchOpc propagate as a ProvisionResult.Failure, allowing retries and error mapping to be handled alongside the rest of the provisioning error handling.

    Step-by-step migration¶

    1. Replace createSdkOutput() callers with an OpcProvider¶

    Anywhere the OPC round-trip was previously invoked manually:

    val sdkOutputResult = cardProvisioning.createSdkOutput()
val sdkOutput = (sdkOutputResult as CreateSdkOutputResult.Success).sdkOutput
val opc = backend.requestOpaquePaymentCardData(paymentInstrumentId, sdkOutput).opc

    …relocate the same backend call into a lambda that the SDK invokes on your behalf:

    val opcProvider = OpcProvider { paymentInstrumentId, sdkOutput ->
    backend.requestOpaquePaymentCardData(paymentInstrumentId, sdkOutput).opc
}

    If OpcProvider is the only argument requiring SAM conversion at the call site, it can also be inlined as a named lambda — see step 2.

    2. Update the provision(...) call site¶

    Before — pre-0.4.0 sequence with manual OPC orchestration:

    // 1. Request sdkOutput from the SDK.
val sdkOutputResult = cardProvisioning.createSdkOutput()
val sdkOutput = (sdkOutputResult as CreateSdkOutputResult.Success).sdkOutput

// 2. Call the backend to exchange sdkOutput for the OPC value.
val opc = backend.requestOpaquePaymentCardData(paymentInstrumentId, sdkOutput).opc

// 3. Pass the OPC back to the SDK to complete provisioning.
val provisionResult = cardProvisioning.provision(
    sdkInput = opc,
    cardDisplayName = "John Doe",
    cardAddress = CardAddress(),
)

    After — single call with an OpcProvider:

    val provisionResult = cardProvisioning.provision(
    cardDisplayName = "John Doe",
    cardAddress = CardAddress(),
    opcProvider = { paymentInstrumentId, sdkOutput ->
        backend.requestOpaquePaymentCardData(paymentInstrumentId, sdkOutput).opc
    },
)

    The example app's ProvisioningSdkRepository demonstrates the same pattern integrated with a real backend client.

    3. Remove the now-unused CreateSdkOutputResult handling¶

    Any when expression that exhausted CreateSdkOutputResult.Success / CreateSdkOutputResult.Failure.* branches can be removed entirely. Failures that previously surfaced there are now reflected in the ProvisionResult returned by the new provision(...) — update the existing ProvisionResult mapping if it does not already cover those cases.

    4. Update the dependency version¶

    Update the published version in the module's build.gradle.kts:

    dependencies {
    implementation("com.adyen.issuing.provisioning:provisioning-sdk:0.5.0-docs-preview")
}

    The deprecated APIs remain on the binary surface, so a transitive consumer that has not yet migrated will still link successfully — but new code on the 0.4.0+ API has no compile-time dependency on createSdkOutput() or the three-argument provision(...).

    What remains unchanged¶

    • CardProvisioning.create(sdkInput, activityProvider) — unchanged. The sdkInput returned by the GET /networkTokenActivationData backend call is still passed at construction time.
    • canProvision() — unchanged signature and CanProvisionResult variants.
    • CardAddress — same package, same fields.
    • Backend endpoints — the two networkTokenActivationData calls are still issued; only the timing of their invocation on the client changes.
